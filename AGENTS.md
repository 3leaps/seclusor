## Project Overview

**seclusor** is a git-trackable secrets management library and CLI using age encryption, implemented in Rust with first-class bindings for Go and TypeScript.

**Architecture**: Library-first — core Rust crates provide all domain logic; CLI and FFI are thin consumers. See `.plans/active/v0.1.0/00-bootstrap-plan.md` for crate decomposition.

**Core differentiator**: Two storage codecs (bundle for safety, inline for git-friendliness) with library-level access for programmatic consumers.

**Primary consumer**: The [Lanyte](https://github.com/lanytehq/lanyte) secure agent platform uses seclusor-crypto and seclusor-keyring as Rust crate dependencies for session attestation (Ed25519 key management encrypted with age). This is the driving reason for the library-first Rust rewrite.

**Sibling repos**: Follows the same Rust + bindings patterns as [ipcprims](https://github.com/3leaps/ipcprims) and [sysprims](https://github.com/3leaps/sysprims).

## Safety Protocols

### Crypto Safety

When modifying encryption or key management code:

- [ ] No plaintext in error messages or logs
- [ ] Key material never written to repo root (pathguard)
- [ ] Size limits enforced before allocation (16 MiB decrypt, 1 MiB inline)
- [ ] Ciphertext prefix validated (`sec:age:v1:`)
- [ ] Identity file permissions checked (0600 on Unix)
- [ ] Passphrase input via rpassword (not command line args)

### FFI Safety

FFI boundary changes require extra scrutiny:

- Memory ownership must be explicit (caller-owned vs library-owned)
- All returned strings freed via `seclusor_free_string()`
- No plaintext secrets in FFI error messages
- Opaque handles for stateful objects
- JSON-over-FFI for complex data returns
- Panic-safe boundary with `catch_unwind`

### Stdout Purity

- Data output → stdout
- Diagnostics, logs, prompts → stderr
- `list` never shows values
- `get` redacts by default (requires `--reveal`)

## DO / DO NOT

### DO

- Run `cargo fmt && cargo clippy` before commits
- Read files before editing them
- Keep changes focused on the task
- Consider library API surface (what's `pub`)
- Consider FFI memory safety implications
- Test with adversarial input for crypto/parsing code
- Document platform-specific behavior

### DO NOT

- Push without maintainer approval
- Skip quality gates or license checks
- Commit secrets or credentials
- Add GPL/LGPL/AGPL dependencies
- Touch code outside task scope without justification
- Change FFI contracts without review
- Use `unsafe` without clear justification and review
- Leak plaintext in error messages or logs
- Print secrets to stdout without explicit `--reveal`
- **EVER commit anything from `.plans/`** — gitignored, MUST stay local
- Commit `AGENTS.local.md` (gitignored — session-specific guidance)

## Critical Rules

### Never Push Without Approval

Git push operations require explicit, per-incident human maintainer approval:

```bash
git add <files>       # OK
git commit -m "..."   # OK
git push              # NEVER without explicit approval
```

### License Compliance

All dependencies must pass `cargo deny check`:

```bash
cargo deny check licenses    # Must pass
cargo deny check advisories  # Must pass
```

Seclusor is dual-licensed MIT OR Apache-2.0. No copyleft dependencies allowed.

### Platform Parity

Changes must consider all supported platforms:

- Linux x64/arm64 (glibc + musl)
- macOS arm64
- Windows x64 (future)
