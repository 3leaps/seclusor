# Release Notes — v0.1.1

**Release Date**: March 2026
**Status**: Post-MVP hardening

## Summary

v0.1.1 extends the v0.1.0 MVP with three significant additions: Ed25519 signing primitives in the Rust library, the first phase of cross-platform Go bindings infrastructure, and a comprehensive test harness for the `secrets run` subcommand. It also contains a security fix for the CI artifact action supply chain.

## What's New

### Ed25519 Signing (`seclusor-crypto/signing`)

`seclusor-crypto` now includes feature-gated Ed25519 signing primitives, enabling Rust consumers to generate keypairs, sign messages, and verify signatures without taking on a second cryptographic dependency.

**Motivation**: The primary driver is `lanyte-attest` (CRT-012), which unseals Ed25519 signing keys encrypted with age and uses them to sign JWT session tokens. Without this, consumers must link both `seclusor-crypto` and a raw `ed25519-dalek` dependency, managing two crypto abstractions with different error models.

**API surface** (all behind `#[cfg(feature = "signing")]`):

| Function                                 | Description                                       |
| ---------------------------------------- | ------------------------------------------------- |
| `generate_signing_keypair()`             | Generate a new Ed25519 keypair from OS randomness |
| `sign(secret_key, message)`              | Sign opaque bytes, return 64-byte signature       |
| `verify(public_key, message, signature)` | Verify a signature                                |
| `signing_public_key(secret_key)`         | Derive public key from secret key                 |
| `signing_secret_key_from_bytes(bytes)`   | Construct from canonical 32-byte seed             |
| `signing_public_key_from_bytes(bytes)`   | Construct from canonical 32-byte public key       |
| `signature_from_bytes(bytes)`            | Construct from canonical 64-byte signature        |
| `signing_secret_key_to_bytes(key)`       | Export 32-byte seed (handle with care)            |
| `signing_public_key_to_bytes(key)`       | Export 32-byte public key                         |
| `signature_to_bytes(sig)`                | Export 64-byte signature                          |

**Security posture**:

- `SigningSecretKey` implements `Zeroize` + `ZeroizeOnDrop`. No `Debug`, `Clone`, or serde — accidental exposure in logs or serialization is a compile error.
- Dalek signing state is derived ephemerally from the stored 32-byte seed. The seed is the canonical key representation.
- Four content-free error variants: `InvalidSecretKeyBytes`, `InvalidPublicKeyBytes`, `InvalidSignatureBytes`, `SignatureVerificationFailed`. No key material, message bytes, or signature bytes ever appear in error strings.
- No X25519/Ed25519 key conversion, no file formats, no JWT framing — signing primitives only.

**Key at rest pattern** (typical consumer usage):

```rust
// Generate and store encrypted
let keypair = seclusor_crypto::generate_signing_keypair()?;
let seed = seclusor_crypto::signing_secret_key_to_bytes(keypair.secret_key());
let encrypted = seclusor_crypto::encrypt(&seed, &recipients)?;
// seed zeroized when it goes out of scope (or zeroize explicitly)

// Load and sign
let decrypted = seclusor_crypto::decrypt(&encrypted, &identities)?;
let sk = seclusor_crypto::signing_secret_key_from_bytes(&decrypted)?;
let sig = seclusor_crypto::sign(&sk, payload)?;
```

**Scope note**: Signing is Rust-native only in v0.1.1. FFI, Go, and TypeScript signing bindings are a deliberate second pass tracked as D11B, targeting v0.1.2. This allows the signing byte contract (fixed in DDR-0002) to be reviewed before it is bound to the stable v0.1.x ABI surface.

**Dependencies added**: `ed25519-dalek 2` (optional, BSD-3-Clause), `zeroize` (optional, MIT/Apache-2.0), `rand_core` (optional, already transitive).

See [ADR-0011](../decisions/ADR-0011-ed25519-signing-in-seclusor-crypto.md) and [DDR-0002](../decisions/DDR-0002-ed25519-signing-contract.md).

### Go Bindings Phase 1

v0.1.1 completes the infrastructure needed for `go get github.com/3leaps/seclusor/bindings/go/seclusor` to work reliably from any platform.

**Platform-specific CGo files (D9B)**: The CGo link directives previously embedded inline in `seclusor.go` have been split into per-platform files (`cgo_linux_amd64.go`, `cgo_linux_arm64.go`, `cgo_darwin_amd64.go`, `cgo_darwin_arm64.go`, `cgo_windows_amd64.go`). Each file specifies `-L${SRCDIR}/lib/local/<platform>` (local dev builds) and `-L${SRCDIR}/lib/<platform>` (CI-built libs) so the linker searches both paths. This matches the ipcprims/sysprims pattern and is required for the CI workflow below.

**`go-bindings.yml` workflow (D9C)**: A manually-dispatched workflow that cross-compiles `seclusor-ffi` for all five supported platforms and opens a pull request with the prebuilt `.a` files committed under `bindings/go/seclusor/lib/<platform>/`. This PR must be merged before each release tag. The workflow uses `cargo-zigbuild` (glibc 2.17 floor) for Linux cross-compilation and native `cargo build` for macOS.

**Supported platforms for Go**:

| Platform      | Build method             | Library path         |
| ------------- | ------------------------ | -------------------- |
| linux/amd64   | zigbuild (glibc 2.17)    | `lib/linux-amd64/`   |
| linux/arm64   | zigbuild (glibc 2.17)    | `lib/linux-arm64/`   |
| darwin/amd64  | native cargo             | `lib/darwin-amd64/`  |
| darwin/arm64  | native cargo             | `lib/darwin-arm64/`  |
| windows/amd64 | MinGW/MSYS2 (GNU target) | `lib/windows-amd64/` |

**Scope of Go bindings in v0.1.1**: Secret document management (`List`, `Get`, `ExportEnv`), bundle encrypt/decrypt, and keyring operations. Ed25519 signing bindings are not included and are planned for v0.1.2.

### `secrets run` Integration Tests (D10)

A comprehensive integration test harness covers the `seclusor secrets run` subcommand: argument parsing, environment injection, `--allow` filtering, subprocess exit code propagation, and edge cases. This completes the run command's testing coverage gap from v0.1.0.

---

### Security Fix: GHA Artifact Actions (D9A)

`actions/upload-artifact` pinned to `v4.6.2` and `actions/download-artifact` pinned to `v4.3.0` across all workflows. Addresses GHSA-cxww-7g56-2vh6 (zip-slip vulnerability in artifact download-artifact prior to v4.1.8 / upload-artifact prior to v4.6.1).

---

## Upgrade Notes

- No breaking changes to the existing API surface. All v0.1.0 code compiles without modification.
- To use Ed25519 signing, add `features = ["signing"]` to the `seclusor-crypto` dependency in `Cargo.toml`.
- Go consumers: the library path layout has changed. If you were building from source, run `make go-bindings-sync` to repopulate `lib/local/<platform>/`.
