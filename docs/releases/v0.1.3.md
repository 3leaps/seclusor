# Release Notes — v0.1.3

**Release Date**: March 30, 2026
**Status**: Error Redaction, Credential Recovery, and Description Metadata

## Summary

v0.1.3 is a security and usability patch. It plugs a secret-leak vector in
JSON error messages, adds actionable credential format validation, introduces
lenient `secrets unset` for repairing malformed files from the CLI, and adds
credential description metadata to the CLI surface.

## Security

### Serde Error Redaction (SC-004 Part A)

When a secrets file contained a malformed credential (e.g., a bare string where
a credential object was expected), serde deserialization errors echoed the
literal plaintext value to stderr and FFI error state. This violated the "no
plaintext in error messages" invariant documented in AGENTS.md and ADR-0006.

**Fixed across all boundaries:**

- `seclusor-core` — `From<serde_json::Error>` now sanitizes via
  `sanitize_serde_json_error_message()` before storing
- `seclusor-codec` — same sanitization at the codec `From` impl
- `seclusor` (CLI) — `CliError::Json` sanitized; `eprintln!` on stderr
  only emits redacted messages
- `seclusor-ffi` — `From<serde_json::Error> for FfiError` sanitized
- `bindings/typescript` — NAPI error reason sanitized

Redaction covers strings, integers, floats, booleans, characters, and byte
arrays. A defense-in-depth fallback sanitizer catches `invalid type:` and
`invalid value:` segments generically, independent of exact serde_json
formatting.

## What's New

### Credential Format Validation (SC-004 Part B)

A custom `Credential` deserializer replaces the derived serde impl. When a
non-object value (string, array, number, etc.) appears where a credential
object is expected, the error now reports:

```
credential "API_KEY" must be an object with "type" plus either "value" or
"ref" string fields, not a string. Use: seclusor secrets set --key API_KEY
--value <value>
```

Key names are structural metadata and safe to display. Values are never
included.

### Lenient `secrets unset` Recovery (SC-004 Part C)

When `secrets unset` encounters a file that fails typed deserialization, it
now falls back to `serde_json::Value`-based manipulation:

1. Parses the file as raw JSON
2. Locates the credential by project slug and key name
3. Removes the entry
4. Writes back valid JSON
5. Attempts strict re-validation and warns if other malformed entries remain

This enables CLI-only repair of files containing malformed credentials
without requiring manual editing (and thus without exposing plaintext secrets).
A stderr warning is emitted when the fallback path activates.

### Credential Description CLI Surface (SC-006)

- `secrets set --description <text>` now stores credential descriptions
- `secrets get --show-description` prints description metadata only
- `secrets list --verbose` prints `KEY<TAB>description` without exposing values
- Description normalization and validation are centralized in
  `seclusor-core`, with runtime and schema rules aligned:
  - credential descriptions are single-line, max 128 characters
  - file/project descriptions are docstrings, max 512 characters, newline allowed
  - Unicode length checks count characters, not UTF-8 bytes

## Notes

- No schema or API contract changes in this release
- The `serde_json::Value` fallback in lenient unset writes the file before
  post-validation; if other malformed entries exist, the targeted entry is
  removed but the command returns an error with an explicit warning
